CHAPTER 00

Setup & the ECS mindset

Install the toolchain, create the project, and rewire how you think about game code.

Before a single sprite appears, two things need to be true: your machine can compile Rust, and your brain has swapped its usual object-oriented model for the one Bevy actually uses — ECS. Get the mindset right now and every later chapter reads as obvious instead of magic.

Install the toolchain

Install Rust via rustup. That gives you rustc (the compiler) and cargo (the build tool, package manager, and test runner in one). Verify:

terminalshell
rustc --version   // rustc 1.8x.x
cargo --version   // cargo 1.8x.x

Bevy compiles a lot of code the first time. That first build is slow (minutes); every build after is fast because Cargo caches dependencies. On Linux you'll also want the usual dev libraries (alsa, udev); Windows and macOS work out of the box.

Create the project

terminalshell
cargo new insta-kill
cd insta-kill
cargo run          // prints "Hello, world!" — the toolchain works

Cargo made you a Cargo.toml (the project manifest) and src/main.rs (the entry point). Those two files are the whole project right now. Everything we build lands under src/.

🦀 Rust concept · Crates & Cargo

A crate is Rust's unit of compilation — roughly, one library or one binary. Your game is a binary crate. Everything you depend on (Bevy, physics, audio) is a library crate downloaded from crates.io. Cargo.toml lists what you depend on; cargo build resolves and compiles the whole tree.

The mental shift: ECS

If you've written games before, you probably reach for inheritance: a Player class extends Character extends Entity, each with methods that own their data and behavior. Bevy throws that out. It uses Entity–Component–System:

Entity → just an ID + Component = data attached to it
System = a plain function that queries components and mutates them

Data lives in components; behavior lives in systems. Nothing owns both.

  • Entity — a bare ID, like a row number. It has no data of its own.
  • Component — a plain struct of data you attach to an entity: Transform (where it is), Sprite (how it looks), your own Player tag.
  • System — an ordinary function. Bevy looks at its parameters, figures out what data it wants, and calls it every frame with exactly that data.

So "the player" is not a class. It's an entity that happens to carry a Player component, a Sprite, a Transform, a physics body, and a few children. A movement system asks Bevy for "every entity with a Velocity and a Player" and nudges them. This is the single most important idea in the whole codebase — the real insta-kill is ~50 small systems, each doing one job to whatever entities match.

🕹 Bevy concept · Why ECS wins for games

Thousands of entities, each a handful of components, processed by tight systems that touch only the data they declare. That's cache-friendly, trivially parallel (Bevy runs non-conflicting systems on different threads for free), and endlessly composable — you add a behavior by adding a component and a system, never by editing a giant class. Keep asking "what component marks this, and what system acts on it?"

CHAPTER 01

Your first window

Add Bevy, open a window, and meet the App builder that assembles the whole game.

Declare the dependency

Open Cargo.toml and add Bevy under [dependencies]. The final game pulls in a lot more; we'll add each one in the chapter that needs it. For now, just Bevy:

Cargo.tomlstart here
[package]
name = "insta-kill"
version = "0.1.0"
edition = "2021"

[dependencies]
bevy = "0.18"

// Compile our own code fast in dev, but optimize the heavy
// dependencies (Bevy, physics) so the game runs smoothly.
[profile.dev]
opt-level = 1

[profile.dev.package."*"]
opt-level = 3
🦀 Rust concept · Profiles

A profile tunes how Cargo compiles. Debug builds are normally unoptimized so they compile fast — but an unoptimized game engine crawls. The trick above (used verbatim in the real project) keeps your code unoptimized and fast to rebuild, while optimizing dependencies once. Best of both.

The smallest Bevy app

Replace src/main.rs entirely:

src/main.rsv1
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .run();
}

cargo run and — after that first long compile — a window opens. Empty, but real: it has a render loop, an event loop, input handling, and asset support already wired up. That's what DefaultPlugins bought you.

🕹 Bevy concept · The App & plugins

App::new() creates an empty application; you configure it with a chain of .add_plugins(...), .add_systems(...), and .insert_resource(...) calls, then .run() starts the loop. A plugin is just a bundle of that configuration. DefaultPlugins is Bevy's bundle of the essentials — windowing, rendering, input, assets, audio. Later, every feature we build becomes its own plugin, and main.rs just lists them.

🦀 Rust concept · use and the prelude

use bevy::prelude::*; pulls Bevy's most-used types into scope so you can write App instead of bevy::app::App. A prelude is a convention — a module that re-exports the things you almost always want. You'll write this line at the top of nearly every file in the project.

Configure the window

The real game customizes the window and how images are sampled. We .set(...) individual plugins inside DefaultPlugins instead of accepting their defaults:

src/main.rsv2
use bevy::prelude::*;
use bevy::window::{PresentMode, Window, WindowMode};

const BACKGROUND_COLOR: Color = Color::srgb(0.75, 0.6, 0.5);

fn main() {
    App::new()
        .add_plugins(
            DefaultPlugins
                .set(WindowPlugin {
                    primary_window: Some(Window {
                        present_mode: PresentMode::Fifo,   // vsync
                        mode: WindowMode::Windowed,
                        fit_canvas_to_parent: true,        // matters for the web build
                        ..default()
                    }),
                    ..default()
                })
                // Pixel art: sample textures with nearest-neighbor, not blurry linear.
                .set(ImagePlugin::default_nearest()),
        )
        .insert_resource(ClearColor(BACKGROUND_COLOR))
        .run();
}

That warm tan, Color::srgb(0.75, 0.6, 0.5), is the real game's background — the whole page you're reading is themed off it. ClearColor is a resource: a single global value (not attached to any entity) that the renderer reads each frame to wipe the screen.

🦀 Rust concept · ..default() and const

Window { ..default() } means "set these fields, fill the rest with default values" — Rust's struct-update syntax, indispensable with Bevy's big config structs. And const defines a compile-time constant; the codebase uses const heavily for tuning values (speeds, offsets, colors) so they sit named at the top of a file instead of buried as magic numbers.

▶ Run it now

cargo run → a window with a solid dusty-tan background. No camera yet, so nothing else renders — that's next. You've built the skeleton the entire game hangs on.

CHAPTER 02

Camera & first sprite

Spawn a 2D camera and put a picture on screen — your first entities, components, and system.

A Bevy scene renders nothing without a camera. And nothing appears without entities to draw. In this chapter you write your first system — a function that spawns entities at startup.

A startup system

src/main.rsadd
fn setup(mut commands: Commands) {
    commands.spawn(Camera2d);
}

// in main(), after the plugins:
// .add_systems(Startup, setup)
🕹 Bevy concept · Systems, schedules & Commands

A system is a function whose parameters tell Bevy what it needs. You register it against a schedule: Startup runs once at launch; Update runs every frame. Commands is your handle for structural changes — spawning and despawning entities, adding components. commands.spawn(Camera2d) creates an entity carrying the Camera2d component, which is all Bevy needs to start rendering a 2D view.

🦀 Rust concept · mut & ownership

mut commands: the parameter is mutable. Rust variables are immutable by default — you opt into mutation with mut. This is the front door to Rust's big idea, ownership: the compiler tracks who can read and who can write each piece of data, which is how it guarantees memory safety and data-race freedom with no garbage collector. You'll see mut on almost every system parameter that changes the world.

Draw a sprite

To draw an image you need an asset: a file loaded at runtime and referenced by a Handle. Put any PNG at assets/player/player.png (Bevy looks in an assets/ folder next to Cargo.toml), then:

src/main.rssetup
fn setup(mut commands: Commands, asset_server: Res<AssetServer>) {
    commands.spawn(Camera2d);

    commands.spawn((
        Sprite::from_image(asset_server.load("player/player.png")),
        Transform::from_scale(Vec3::splat(2.0)),
    ));
}

Note the double parentheses: spawn((A, B)) spawns one entity with a tuple of components. Here the entity gets a Sprite (what to draw) and a Transform (where, rotated how, scaled how much). Vec3::splat(2.0) just means (2.0, 2.0, 2.0) — the game renders its pixel art at 2× so it reads at a comfortable size.

🕹 Bevy concept · Assets & handles

asset_server.load(...) kicks off loading a file on a background thread and immediately returns a Handle<Image> — a cheap, clonable reference. The sprite shows a blank frame until the bytes arrive, then updates automatically. Res<AssetServer> is a resource parameter: shared read-only access to that global loader. In the next chapter we replace this ad-hoc load with a disciplined, preloaded asset collection.

▶ Run it now

Your image sits centered on the tan background. Move the camera or the sprite by changing the Transform translation and watch it shift. You now have entities, components, a system, and the asset pipeline — the whole ECS loop in miniature.

CHAPTER 03

States & loading assets

Add a game-state machine and load every asset up front so systems can assume it's ready.

Loading assets ad-hoc (like chapter 2) is fine for one image and miserable for a hundred. The real game loads everything during a dedicated loading state, then switches to gameplay only once it's all in memory. This is the backbone that lets every later system write assets.player.clone() and trust it exists.

Define the states

src/main.rsGameState
#[derive(States, Clone, Eq, PartialEq, Debug, Hash, Default)]
pub enum GameState {
    #[default]
    AssetLoading,
    Gaming,
    GameOver,
    Restart,
}
🦀 Rust concept · Enums & derive

A Rust enum is a type that is exactly one of a fixed set of variants — perfect for a state machine. #[derive(...)] auto-generates trait implementations: here Clone, equality, hashing, debug-printing, and Bevy's States trait. Deriving is how you get a lot of behavior for free without writing boilerplate — you'll #[derive(Component)] and #[derive(Resource)] constantly.

🕹 Bevy concept · States

A state is a global enum resource Bevy watches. You register it with .init_state::<GameState>(). Then you can gate systems: run this only in_state(GameState::Gaming), run that once OnEnter(GameState::GameOver). Almost every gameplay system in the project is gated on Gaming, so nothing runs while assets are still loading or the game-over screen is up.

Bring in bevy_asset_loader

A community crate, bevy_asset_loader, turns "load a pile of files, then advance a state when they're all done" into a declarative struct. Add it:

Cargo.tomladd
bevy_asset_loader = { version = "0.25.0", features = ["2d"] }

Now describe your assets as a single Resource. Each field is one asset; the attribute above it says where it comes from. This is a trimmed slice of the real src/assets.rs:

src/assets.rsGameAssets

🕹 Bevy concept · Texture atlases

Sprite-sheet animation packs every frame into one image. A TextureAtlasLayout is the grid definition — "80×80 tiles, 23 columns, 5 rows" — and a sprite picks a frame by index into that grid. The player sheet has 23×5 = 115 frames covering idle, running, dashing, and more. We'll flip through those indices to animate in the next chapter.

Wire it into the App

src/main.rsloading

Read it top to bottom: start in AssetLoading, load every field of GameAssets, and the instant they're all ready, switch to Gaming. From that point on, any system gated on Gaming can take assets: Res<GameAssets> and know every handle is loaded.

Nothing visible changes yet — but the architecture just leveled up. You have a state machine and a guaranteed-loaded asset bundle. Every remaining chapter spawns things using GameAssets and runs OnEnter(GameState::Gaming).

CHAPTER 04

The animation system

Write a custom asset loader for a tiny animation format, and a component that flips atlas frames over time.

Bevy has no built-in sprite-sheet animation player, so insta-kill ships its own — about 200 lines in src/animation.rs. It's the most "systems programming" chapter in the whole tutorial and the payoff is huge: a clean animator.play(clip).repeat() API used everywhere. We'll build it in three pieces: a clip format, an asset loader, and a player component.

The clip format (.trickfilm)

An animation is just "which frames, over how long." The game stores that as a small RON file (Rust Object Notation) — one .trickfilm file can hold several named clips:

assets/player/player.trickfilmdata
[
    (
        name: "idle",
        keyframes: KeyframesRange((start: 0, end: 9)),
        duration: 0.9,
    ),
    (
        name: "moving",
        keyframes: KeyframesRange((start: 23, end: 29)),
        duration: 0.5,
    ),
    (
        name: "hooking",
        keyframes: KeyframesVec([94]),   // a single, specific frame
        duration: 1.0,
    ),
]

Two ways to specify frames: a contiguous start..end range, or an explicit list. We model both in Rust and load the file into typed clips.

src/animation.rstypes
use serde::Deserialize;

#[derive(Debug, Clone, Deserialize)]
struct KeyframeRange { start: usize, end: usize }

#[derive(Debug, Clone, Deserialize)]
enum Keyframes {
    KeyframesRange(KeyframeRange),
    KeyframesVec(Vec<usize>),
}

impl Keyframes {
    fn into_indices(self) -> Vec<usize> {
        match self {
            Keyframes::KeyframesRange(r) => (r.start..r.end).collect(),
            Keyframes::KeyframesVec(v) => v,
        }
    }
}

#[derive(Debug, Clone, Deserialize)]
struct AnimationClipDef { name: String, keyframes: Keyframes, duration: f32 }

// The runtime clip: a flat list of frame indices + how long to play them.
#[derive(Asset, TypePath, Debug, Clone)]
pub struct AnimationClip2D {
    keyframes: Vec<usize>,
    duration: f32,
}

impl AnimationClip2D {
    fn frame_at(&self, seek_time: f32) -> usize {
        let len = self.keyframes.len();
        let idx = ((seek_time / self.duration) * len as f32) as usize;
        self.keyframes[idx.min(len - 1)]
    }
}
🦀 Rust concept · Enums with data, match, and serde

An enum variant can carry data — Keyframes is either a range or a vector, and match forces you to handle every case, giving each its value. Deriving Deserialize (from the serde crate) auto-generates code to parse the RON text straight into these structs — no manual parsing. This "describe the shape, get parsing for free" pattern is everywhere in Rust.

Teaching Bevy to load it

To make .trickfilm files first-class assets you implement Bevy's AssetLoader trait: read the bytes, parse them, and register each named clip as a labeled sub-asset (so player.trickfilm#idle becomes a handle).

src/animation.rsloader
use bevy::asset::{AssetLoader, LoadContext, io::Reader};
use thiserror::Error;

#[derive(Debug, Error)]
pub enum AnimationClip2DLoaderError {
    #[error("could not read animation manifest: {0}")]
    Io(#[from] std::io::Error),
    #[error("could not parse animation manifest: {0}")]
    Ron(#[from] ron::error::SpannedError),
}

#[derive(Default, TypePath)]
struct AnimationClip2DLoader;

impl AssetLoader for AnimationClip2DLoader {
    type Asset = AnimationClip2D;
    type Settings = ();
    type Error = AnimationClip2DLoaderError;

    async fn load(
        &self,
        reader: &mut dyn Reader,
        _settings: &Self::Settings,
        load_context: &mut LoadContext<'_>,
    ) -> Result<Self::Asset, Self::Error> {
        let mut bytes = Vec::new();
        reader.read_to_end(&mut bytes).await?;
        let defs: Vec<AnimationClipDef> = ron::de::from_bytes(&bytes)?;

        let mut first = None;
        for def in defs {
            let clip = AnimationClip2D {
                keyframes: def.keyframes.into_indices(),
                duration: def.duration,
            };
            // Register each clip under its name, e.g. "...#idle".
            load_context.add_labeled_asset(def.name, clip.clone());
            if first.is_none() { first = Some(clip); }
        }
        Ok(first.expect("manifest must contain at least one clip"))
    }

    fn extensions(&self) -> &[&str] { &["trickfilm"] }
}
🦀 Rust concept · Traits, Result, and the ? operator

A trait is a set of methods a type promises to provide — implementing AssetLoader for AnimationClip2DLoader is how you plug into Bevy. Fallible work returns Result<T, E> (either Ok(value) or Err(error)), and the ? operator says "unwrap the Ok, or return the Err early." The #[from] on the error enum (via the thiserror crate) lets ? auto-convert an io::Error or RON error into our error type. This is Rust's entire error-handling story — no exceptions, just values.

The player component that drives frames

Finally, a component you attach to any animated entity. It remembers the current clip and how far into it we are; one system advances every animator each frame and writes the resulting frame index into the sprite's atlas.

src/animation.rsplayer + system
#[derive(Component)]
pub struct AnimationPlayer2D {
    clip: Handle<AnimationClip2D>,
    seek_time: f32,
    speed: f32,
    repeat: bool,
    finished: bool,
    // ...a couple more flags in the real file (reverse, just_finished_cycle)
}

// We write Default by hand instead of deriving it: a *derived* Default sets
// speed = 0.0, and animate_sprites advances by `delta * speed` — so the clip
// would sit frozen on frame 0 forever. We want forward playback out of the box.
impl Default for AnimationPlayer2D {
    fn default() -> Self {
        Self {
            clip: Handle::default(),
            seek_time: 0.0,
            speed: 1.0,
            repeat: false,
            finished: false,
        }
    }
}

impl AnimationPlayer2D {
    pub fn play(&mut self, clip: Handle<AnimationClip2D>) -> &mut Self {
        // Guard: if we're already playing this clip, don't restart it.
        // State systems call play() every frame with the current state's clip;
        // without this the animation would reset to frame 0 forever.
        if self.clip == clip { return self; }
        self.clip = clip;
        self.seek_time = 0.0;
        self.finished = false;
        self
    }
    pub fn repeat(&mut self) -> &mut Self { self.repeat = true; self }

    // Read-only accessor: chapter 7's `leave_dash` uses this to know when a
    // one-shot clip (like the dash) has played out.
    pub fn is_finished(&self) -> bool { self.finished }
}

fn animate_sprites(
    time: Res<Time>,
    clips: Res<Assets<AnimationClip2D>>,
    mut q: Query<(&mut AnimationPlayer2D, &mut Sprite)>,
) {
    for (mut player, mut sprite) in &mut q {
        if player.finished && !player.repeat { continue; }
        let Some(clip) = clips.get(&player.clip) else { continue; };
        let Some(atlas) = sprite.texture_atlas.as_mut() else { continue; };

        player.seek_time += time.delta_secs() * player.speed;
        if player.seek_time >= clip.duration {
            if player.repeat { player.seek_time -= clip.duration; }
            else { player.finished = true; player.seek_time = clip.duration; }
        }
        atlas.index = clip.frame_at(player.seek_time);  // the actual animation!
    }
}
🕹 Bevy concept · Queries & Res<Time>

A Query is how a system asks for entities: Query<(&mut AnimationPlayer2D, &mut Sprite)> means "give me every entity that has both, and let me mutate both." Iterating the query touches exactly those entities. Res<Time> is Bevy's clock; time.delta_secs() is the seconds since last frame, which is how you make motion frame-rate independent — always scale by delta time, never assume a fixed frame rate.

🦀 Rust concept · let ... else & Option

clips.get(...) returns an Option — either Some(clip) or None (still loading). The let Some(clip) = ... else { continue; }; form unwraps the happy path and bails cleanly on None. Rust has no null; absence is a value you're forced to handle, which is why whole categories of crashes simply don't exist here.

Put the player on screen

Three things have to come together, and each lives in a file we've already met. First, teach GameAssets about the animation clips. Back in chapter 3 we loaded the player image and its atlas layout; now load the animation clips as one ordered Vec. Because OnEnter(Gaming) and OnExit(AssetLoading) then never fire, you get a blank window — no camera, no player — often with no obvious error. If the screen is blank, suspect a missing asset file or a mistyped #label first.

Next, a system that spawns the player entity. This is where the engine and the assets meet: a Sprite that points at the sheet image and carries a TextureAtlas (the layout + which frame to show), plus an AnimationPlayer2D set to play the idle clip on a loop.

src/main.rssetup_player
fn setup_player(mut commands: Commands, assets: Res<GameAssets>) {
    let mut animator = AnimationPlayer2D::default();
    animator.play(assets.player_animations[0].clone()).repeat();  // 0 = idle

    commands.spawn((
        Sprite {
            image: assets.player.clone(),
            texture_atlas: Some(TextureAtlas {
                layout: assets.player_layout.clone(),
                index: 0,
            }),
            ..default()
        },
        Transform::from_xyz(0.0, 0.0, 0.0),
        animator,
    ));
}
🕹 Bevy concept · An animated sprite is three cooperating pieces

The image is the pixels, the atlas is the grid that carves those pixels into frames (and holds the current index), and the animator is what changes that index over time. Our animate_sprites system from earlier does the last part: it reads the clip and writes atlas.index every frame. Remove any one of the three and you get a blank screen or a frozen picture — which is exactly why AnimationPlayer2D::default() gives speed = 1.0 rather than 0.0.

Finally, run both systems — the camera and the player — the moment loading finishes. In chapter 2 the camera spawned at Startup; move it onto OnEnter(GameState::Gaming) so it and the player appear together, after GameAssets is guaranteed loaded:

src/main.rswire it up
mod animation;   // NEW FILE — declare it or main.rs can't see Animation2DPlugin

fn setup(mut commands: Commands) {
    commands.spawn(Camera2d);
}

// inside main(), with the other builder calls:
.add_plugins(Animation2DPlugin)
.init_state::<GameState>()
.add_systems(OnEnter(GameState::Gaming), (setup, setup_player))
🦀 Rust concept · A new .rs file needs a mod

Creating src/animation.rs is not enough — Rust only compiles files it's been told about. The mod animation; line in main.rs declares it (just like mod assets; back in chapter 3); without it you'll get "cannot find Animation2DPlugin in this scope". We cover how modules nest into folders in chapter 5 — for the full rules see the Rust Book, Managing Growing Projects.

🦀 Rust concept · OnEnter vs Startup

Gameplay entities need their assets to exist. Startup runs once at launch — before bevy_asset_loader has read a single file — so a sprite spawned there would reach for a handle that isn't ready. OnEnter(GameState::Gaming) runs the instant the loading state finishes and flips to Gaming, so every assets.*.clone() is safe. That's the whole payoff of the state machine we built in chapter 3.

CHAPTER 05

Input as a resource

Collect raw keyboard/mouse input into one tidy PlayerInput resource that the rest of the game reads.

Movement, dashing, striking, and hooking all need to know "what did the player press?" Instead of each system polling the keyboard directly, insta-kill funnels raw input into a single PlayerInput resource once per frame. Gameplay systems then read that clean struct. This decoupling is a small idea with big payoff — it makes rebinding, gamepads, and disabling input trivial.

The input resource

Fill it each frame

One system reads the WASD/mouse state and writes the resource. Note the game supports two key sets (WASD and a Vim-like JKFD layout), and normalizes diagonal movement so it isn't faster:

src/player/input.rssystems
fn reset_player_input(mut player_input: ResMut<PlayerInput>) {
    *player_input = PlayerInput::default();   // clear before we refill
}

fn player_movement(keys: Res<ButtonInput<KeyCode>>, mut input: ResMut<PlayerInput>) {
    let mut direction = Vec2::default();
    if keys.pressed(KeyCode::KeyJ) || keys.pressed(KeyCode::KeyS) { direction.y -= 1.0; }
    if keys.pressed(KeyCode::KeyK) || keys.pressed(KeyCode::KeyW) { direction.y += 1.0; }
    if keys.pressed(KeyCode::KeyF) || keys.pressed(KeyCode::KeyD) { direction.x += 1.0; }
    if keys.pressed(KeyCode::KeyA)                                { direction.x -= 1.0; }
    input.move_direction = direction.normalize_or_zero();
}

fn attack(mouse: Res<ButtonInput<MouseButton>>, mut input: ResMut<PlayerInput>) {
    input.attack = mouse.just_pressed(MouseButton::Left);
}

fn dash(keys: Res<ButtonInput<KeyCode>>, mut input: ResMut<PlayerInput>) {
    input.dash = keys.just_pressed(KeyCode::Space);
}

fn hook(keys: Res<ButtonInput<KeyCode>>, mut input: ResMut<PlayerInput>) {
    input.hook = keys.just_pressed(KeyCode::ShiftLeft);   // read by the hook in ch11
}

fn input_escape(keys: Res<ButtonInput<KeyCode>>, mut input: ResMut<PlayerInput>) {
    input.escape = keys.just_pressed(KeyCode::Escape);    // game-over screen, ch19
}

fn restart(keys: Res<ButtonInput<KeyCode>>, mut input: ResMut<PlayerInput>) {
    input.restart = keys.just_pressed(KeyCode::KeyR);     // game-over screen, ch19
}

// The scroll wheel feeds the master volume (ch17). MouseWheel is a *message*, not a
// button, so read it with a MessageReader — add `use bevy::input::mouse::MouseWheel;`.
fn fetch_scroll_events(mut scroll_evr: MessageReader<MouseWheel>, mut input: ResMut<PlayerInput>) {
    for ev in scroll_evr.read() {
        input.scroll = if ev.y > 0.0 { -1.0 } else { 1.0 };
    }
}

Every field of PlayerInput now has a system that fills it — miss one and the corresponding feature (hook, volume, restart) silently never fires, since the field just stays at its default.

🕹 Bevy concept · Res vs ResMut, pressed vs just_pressed

Res<T> is shared read access to a resource; ResMut<T> is exclusive write access — the same borrow rules as everywhere in Rust, enforced across systems. ButtonInput distinguishes pressed() (held down this frame, good for movement) from just_pressed() (the single frame it went down, good for actions like attack or dash). Getting that distinction right is the difference between one strike and a hundred.

🦀 Rust concept · Deref with *

*player_input = PlayerInput::default(); — ResMut is a smart pointer wrapping your resource; the * dereferences it to overwrite the value it points to. normalize_or_zero() is a nice safety detail: it scales a direction vector to length 1 (so diagonal isn't √2 faster) but returns zero instead of NaN when there's no input.

Ordering matters

The plugin runs input collection in PreUpdate, after Bevy's own input systems have recorded this frame's raw events, and clears the resource before them. So every gameplay system in Update reads a fully-populated, current-frame snapshot.

    fn build(&self, app: &mut App) {
        app.init_resource::<PlayerInput>()
            .add_systems(PreUpdate, reset_player_input.before(InputSystems))
            .add_systems(
                PreUpdate,
                (player_movement, attack, dash, hook, input_escape, restart, fetch_scroll_events)

    }

🕹 Bevy concept · System ordering

By default systems run in parallel with no guaranteed order. When order matters you say so: .before(x), .after(x), or group with .chain(). Here InputSystems is a built-in label for Bevy's input processing, so .after(InputSystems) means "run once the raw keyboard state is up to date." Sprinkle ordering only where correctness needs it — over-ordering kills the free parallelism.

fn spawn_player(mut commands: Commands, assets: Res<GameAssets>) {
    let mut animator = AnimationPlayer2D::default();
    animator.play(assets.player_animations[0].clone()).repeat();  // idle

    let collider = commands

            Transform::from_translation(PLAYER_SPAWN_POS).with_scale(Vec3::splat(2.0)),

use bevy::prelude::*;
use bevy_rapier2d::prelude::*;               // Velocity
use crate::GameState;
use super::input::PlayerInput;
use super::{Player, PlayerState, MOVE_SPEED};   // constants live in mod.rs

fn player_movement(
    mut q_player: Query<(&mut Velocity, &mut Player)>,
    player_input: Res<PlayerInput>,
) {
    let (mut velocity, mut player) = match q_player.single_mut() {
        Ok(r) => r,
        Err(_) => return,          // no player yet — nothing to do
    };

    // Abilities (dash, hook) own their own states; movement only drives Idling/Moving.
    if player.state != PlayerState::Moving && player.state != PlayerState::Idling {
        return;
    }

    let direction = player_input.move_direction;
    if direction == Vec2::ZERO {
        player.state = PlayerState::Idling;
        velocity.linvel = Vec2::ZERO;
        return;
    }

    player.state = PlayerState::Moving;
    player.current_direction = direction;
    velocity.linvel = direction * MOVE_SPEED;   // physics does the rest
}

Player state machine

Make one PlayerState enum the single source of truth, and let it drive animation, facing, and every ability.

In chapter 6 the movement system quietly set player.state = PlayerState::Moving. That field is about to become the spine of the whole player. Instead of each ability guessing what the player is doing, everyone reads and writes one enum — and a message announces every transition so systems can react exactly once when something changes.

The change message

We already defined PlayerState and the Player component in chapter 6. Now we grow that same src/player/state.rs file: first a message that announces every transition, then the systems that produce and consume it. The two new use lines join the use bevy::prelude::*; already at the top of the file.

🕹 Bevy concept · Messages (events)

A message is a one-shot value one system writes and others read, decoupling sender from receiver. You register it with .add_message::<PlayerChangedState>(), write with a MessageWriter, and read with a MessageReader. The dash and hook systems (next two chapters) don't poll the player's state every frame — they wait for a PlayerChangedState with new_state == Dashing and act on that single edge. (Bevy called these "events" before 0.18; the mechanism is the same.)

Detecting the transition

One system compares the current state against what it was last frame and fires the message. It remembers the previous value with a Local — per-system private state that survives between frames.

src/player/state.rsdetect
fn player_changed_state(
    q_player: Query<&Player>,
    mut ev_changed_state: MessageWriter<PlayerChangedState>,
    mut old_state: Local<PlayerState>,
) {
    let player = match q_player.single() {
        Ok(r) => r,
        Err(_) => return,
    };

    if player.state != *old_state {
        ev_changed_state.write(PlayerChangedState {
            old_state: *old_state,
            new_state: player.state,
        });
        *old_state = player.state;
    }
}
🦀 Rust concept · Copy types & Local

Because PlayerState derives Copy, assigning or comparing it duplicates the value bit-for-bit — no ownership move, no .clone(). That's why *old_state = player.state; just works. Small, cheap enums and numbers are almost always Copy; anything owning a heap allocation (like String or Vec) is not.

State drives the animation

Now the payoff from chapter 4. A system maps each state to its clip and asks the animator to play it. Remember the play() guard — it ignores a request for the clip already playing, so calling this every frame is free.

src/player/state.rsanimate
fn update_animations(
    assets: Res<GameAssets>,
    mut q_player: Query<(&Player, &mut AnimationPlayer2D)>,
) {
    let (player, mut animator) = match q_player.single_mut() {
        Ok(r) => r,
        Err(_) => return,
    };

    let animation = match player.state {
        PlayerState::Idling  => assets.player_animations[0].clone(),
        PlayerState::Moving  => assets.player_animations[1].clone(),
        PlayerState::Dashing => assets.player_animations[2].clone(),
        PlayerState::Hooking => assets.player_animations[3].clone(),
        PlayerState::Sliding => assets.player_animations[4].clone(),
    };
    animator.play(animation).repeat();
}

fn adjust_sprite_flip(mut q_player: Query<(&mut Sprite, &Player)>) {
    let (mut sprite, player) = match q_player.single_mut() {
        Ok(r) => r,
        Err(_) => return,
    };
    // Don't flip mid-dash/hook; otherwise face the travel direction.
    if player.state == PlayerState::Hooking || player.state == PlayerState::Sliding { return; }
    if player.current_direction.x == 0.0 { return; }
    sprite.flip_x = player.current_direction.x < 0.0;
}
🦀 Rust concept · Exhaustive match

The compiler requires the match to cover every PlayerState variant — leave one out and it won't build. So the day you add a sixth state, the compiler marches you to every place that needs updating. This "make illegal states unrepresentable, then let the compiler find the gaps" workflow is a big reason Rust game code refactors so safely.

Leaving a state automatically

Some states end themselves. Dashing lasts exactly as long as the dash animation; hooking ends after a fixed time. These systems watch the animator and drop back to Idling:

fn leave_dash(mut q_player: Query<(&mut Player, &AnimationPlayer2D)>) {
    let (mut player, animator) = match q_player.single_mut() {
        Ok(r) => r, Err(_) => return,

🕹 Bevy concept · PostUpdate ordering

These systems run in PostUpdate — after gameplay in Update has settled this frame's state — and leave_dash is ordered .after(update_animations) so it checks a freshly-advanced animator. CHAPTER 08

Camera feel & Y-sort

Three tricks that make a flat top-down scene feel alive: a leading camera, trauma-based screen shake, and depth sorting.

Nothing here changes what the player can do — it changes how the game feels. A camera that anticipates motion, a screen that kicks when things die, and sprites that overlap correctly are the difference between a tech demo and a game.

A camera that leads

Following the player's exact position feels laggy. The real camera targets slightly ahead of the player, in the direction they're moving, by adding a fraction of their velocity:

}

Screen shake with "trauma"

The classic technique (from a well-known GDC talk): keep a single trauma value in 0–1, bump it when something impactful happens, and decay it every frame. The actual shake is trauma² times some noise — squaring makes small trauma barely wiggle and big trauma slam.

src/world/camera_shake.rsshake
use chrono::Utc;
use bevy::prelude::*;
use bevy_rapier2d::plugin::RapierTransformPropagateSet;
use noisy_bevy::simplex_noise_2d_seeded;

use super::MainCamera;   // re-exported from world::camera at the module root

const NOISE_STRENGTH: f32 = 10.0;
const TRANSLATION_SHAKE_STRENGTH: f32 = 15.0;
const ROTATION_SHAKE_STRENGTH: f32 = 2.5;

#[derive(Resource, Default, Reflect)]
pub struct CameraShake {
    trauma: f32,
    seed: f32,
    target: Vec2,
}

impl CameraShake {
    pub fn add_trauma(&mut self, trauma: f32) {
        if self.trauma == 0.0 {
            self.seed = (Utc::now().timestamp_millis() & 0xFFFF) as f32;
        }
        self.trauma = (self.trauma + trauma.abs()).min(1.0);   // clamp to 1
    }

    pub fn update_target(&mut self, target: Vec2) {
        self.target = target;
    }

    fn reduce_trauma(&mut self, delta: f32) {
        self.trauma = (self.trauma - delta.abs()).max(0.0);    // never below 0
    }

    // One noise sample per axis; `stack` shifts the seed so x/y/rotation decorrelate.
    fn noise_value(&self, stack: u32) -> f32 {
        simplex_noise_2d_seeded(
            Vec2::new(self.trauma * NOISE_STRENGTH, 0.0),
            self.seed + stack as f32,
        )
    }
}

fn decay_shake_trauma(time: Res<Time>, mut shake: ResMut<CameraShake>) {
    shake.reduce_trauma(time.delta_secs());
}

pub fn update_camera(
    mut q_camera: Query<&mut Transform, With<MainCamera>>,
    shake: ResMut<CameraShake>,
) {
    let mut transform = match q_camera.single_mut() { Ok(t) => t, Err(_) => return };

    let translation_offset = Vec3::new(shake.noise_value(0), shake.noise_value(1), 0.0)
        * shake.trauma.powi(2) * TRANSLATION_SHAKE_STRENGTH;      // trauma² feel
    let rotation_offset = Quat::from_rotation_z(
        (shake.noise_value(2) * shake.trauma.powi(2) * ROTATION_SHAKE_STRENGTH).to_radians(),
    );

    transform.translation = shake.target.extend(transform.translation.z) + translation_offset;
    transform.rotation = rotation_offset;
}

pub struct CameraShakePlugin;

impl Plugin for CameraShakePlugin {
    fn build(&self, app: &mut App) {
        app.init_resource::<CameraShake>()
            .add_systems(Update, decay_shake_trauma)
            .add_systems(PostUpdate, update_camera.before(RapierTransformPropagateSet));
    }
}

Faking depth: Y-sort

In a top-down game, something lower on screen should draw in front of something higher — it's "closer" to the camera. Bevy decides draw order by the sprite's z. So we compute z from y: lower y → higher z → drawn on top. Give an entity a YSort component and one system does the rest.

src/world/camera.rsy-sort
pub const TRANSLATION_TO_PIXEL: f32 = 0.0001;

#[derive(Component)]
pub struct YSort(pub f32);   // the base layer; children pass e.g. -1.0 or 1.0

pub fn apply_y_sort(mut q: Query<(&mut Transform, &GlobalTransform, &YSort)>) {
    for (mut transform, global, ysort) in &mut q {
        transform.translation.z = ysort.0 - global.translation().y * TRANSLATION_TO_PIXEL;
    }
}
enemy at y = 300 → smaller z → drawn behind
player at y = 100 → larger z → drawn in front

The shadow child uses YSort(-1.0) so it always sits under its owner.

🦀 Rust concept · Tuple structs

struct YSort(pub f32); is a tuple struct — a named wrapper around one value, accessed as ysort.0. It's a zero-cost way to give a bare f32 a meaningful type so Bevy can query for it as a component. The codebase uses this pattern constantly ( DashTimer(Timer), Seed(u32)) to turn primitives into queryable, distinct types.

🕹 Bevy concept · GlobalTransform & run-before ordering

An entity's Transform is relative to its parent; its GlobalTransform is the resolved world position Bevy computes each frame. Y-sort reads the global y but writes the local z. Both apply_y_sort and the camera update run in PostUpdate, ordered .before(RapierTransformPropagateSet) so the physics engine sees the final transforms — ordering against a named system set, not a single system.

Strike & the hit chain

Aim-at-mouse melee: a message-driven attack, short-lived hit colliders, and a satisfying three-swing combo.

The strike is the core verb of the game. Click, and a slash animation with a hitbox appears aimed at the cursor. Click again quickly and you chain — swing 1, swing 2, then a heavier swing 3 that forces a brief cooldown. This chapter is a masterclass in timers and message-driven spawning.

First, a shared helper module

The strike faces the cursor, which needs two helpers used all over the game: quat_from_vec2 (turn a direction into a Z-rotation) and a FixedRotation component (hold a child's world rotation independent of its parent, so the slash points at the mouse even as the player sprite flips). They live in a new crate-level module, src/utils/mod.rs:

src/utils/mod.rshelpers
use bevy::prelude::*;
use bevy_rapier2d::{plugin::RapierTransformPropagateSet, prelude::*};

pub const COLLISION_GROUPS_NONE: CollisionGroups = CollisionGroups::new(Group::NONE, Group::NONE);

pub fn quat_from_vec2(direction: Vec2) -> Quat {
    if direction == Vec2::ZERO {
        return Quat::IDENTITY;
    }
    Quat::from_euler(EulerRot::XYZ, 0.0, 0.0, Vec2::X.angle_to(direction))
}

pub fn quat_from_vec3(direction: Vec3) -> Quat {
    quat_from_vec2(direction.truncate())   // used by the archer's aim in ch14
}

#[derive(Component, Default)]
pub struct FixedRotation {
    pub offset: Vec3,
    pub rot: Quat,
}

// Cancel the parent's rotation so a child sprite can hold a fixed world rotation.
fn reset_rotations(
    mut q_transforms: Query<(&ChildOf, &mut Transform, &FixedRotation)>,
    q_parents: Query<&Transform, Without<FixedRotation>>,
) {
    for (parent, mut transform, fixed_rotation) in q_transforms.iter_mut() {
        if let Ok(parent_transform) = q_parents.get(parent.parent()) {
            let rot_inv = parent_transform.rotation.inverse();
            transform.rotation = Quat::from_rotation_z(
                rot_inv.to_euler(EulerRot::ZYX).0 + fixed_rotation.rot.to_euler(EulerRot::ZYX).0,
            );
            transform.translation = rot_inv.mul_vec3(fixed_rotation.offset);
        }
    }
}

Two systems, one message

The design splits cleanly: one system reads input, checks cooldowns, and decides to strike by writing a SpawnStrike message; another system listens and spawns the visual + collider. This is the recurring pattern for anything the player creates.

src/player/strike.rstrigger
use std::time::Duration;
use bevy::prelude::*;
use bevy_rapier2d::prelude::*;
use crate::animation::AnimationPlayer2D;
use crate::utils::{quat_from_vec2, FixedRotation};
use crate::world::camera::YSort;
use crate::GameAssets;
use super::input::{MouseWorldCoords, PlayerInput};
use super::Player;

const OFFSET: Vec3 = Vec3::new(0.0, -10.0, 0.0);
const CHAIN_COOLDOWN: f32 = 0.35;      // max gap before the combo resets
const STRIKE_COOLDOWN: f32 = 0.4;      // forced pause after the 3rd swing
const STRIKE_CHAIN_COUNT: usize = 3;   // swings per combo

#[derive(Resource, Default)]
struct StrikeCooldown {
    absolute_cooldown: Timer,   // the post-combo lockout
    chain_cooldown: Timer,      // the "keep combo alive" window
    strike_index: usize,        // 0, 1, 2 -> which swing we're on
}

#[derive(Message)]
pub struct SpawnStrike { pub rot: Quat, strike_index: usize }

fn trigger_strike(
    player_input: Res<PlayerInput>,
    mouse_coords: Res<MouseWorldCoords>,
    mut strike_cooldown: ResMut<StrikeCooldown>,
    q_player: Query<&Transform, With<Player>>,
    mut ev_spawn_strike: MessageWriter<SpawnStrike>,
) {
    let player_transform = match q_player.single() { Ok(r) => r, Err(_) => return };
    if !strike_cooldown.absolute_cooldown.is_finished() { return; }   // locked out

    let rot = quat_from_vec2(mouse_coords.0 - player_transform.translation.truncate());
    if !player_input.attack { return; }

    ev_spawn_strike.write(SpawnStrike { rot, strike_index: strike_cooldown.strike_index });
    strike_cooldown.strike_index += 1;

    strike_cooldown.chain_cooldown.set_duration(Duration::from_secs_f32(CHAIN_COOLDOWN));
    strike_cooldown.chain_cooldown.reset();

    if strike_cooldown.strike_index == STRIKE_CHAIN_COUNT {       // finished the combo
        strike_cooldown.strike_index = 0;
        strike_cooldown.absolute_cooldown.set_duration(Duration::from_secs_f32(STRIKE_COOLDOWN));
        strike_cooldown.absolute_cooldown.reset();
    }
}
🕹 Bevy concept · Timer

A Timer counts elapsed time toward a duration; you advance it each frame with timer.tick(time.delta()) and query is_finished() or just_finished(). A TimerMode::Once timer stops when done and is reused with reset(). Here two timers cooperate: chain_cooldown keeps the combo alive if you keep clicking, and absolute_cooldown enforces the pause after the third hit. A separate tiny system just ticks them both.

🦀 Rust concept · Field privacy

Notice strike_index in the message has no pub — it's private to this module, while rot is public. Rust's default is private; you expose exactly what other modules need. The listener below lives in the same file, so it can read the private field, but nothing outside strike.rs can construct a half-built SpawnStrike. Privacy is how each feature module guards its own invariants.

    assets: Res<GameAssets>,
    q_player: Query<Entity, With<Player>>,

☠ Design note · Why timers everywhere

Almost every transient thing — a swing, its hitbox, a dash effect — is spawned then torn down by a Timer or an animation finishing, never left to accumulate. In a game spawning hundreds of entities a second, disciplined despawning is what keeps it fast and correct. If you ever see the game slow down over time, a missing despawn is the first suspect.

src/player/dash.rslaunch
fn move_player(
    mouse_coords: Res<MouseWorldCoords>,
    mut q_player: Query<(&mut Transform, &mut Velocity), With<Player>>,
    mut ev_changed_state: MessageReader<PlayerChangedState>,
) {
    let (mut transform, mut velocity) = match q_player.single_mut() {
        Ok(r) => r, Err(_) => return,
    };

    for ev in ev_changed_state.read() {
        if ev.new_state != PlayerState::Dashing { continue; }

        let dir = (mouse_coords.0 - transform.translation.truncate()).normalize_or_zero();
        transform.rotation = quat_from_vec2(dir);       // face the dash
        velocity.linvel = dir * DASH_MULTIPLIER * MOVE_SPEED;
    }
}
🕹 Bevy concept · Reading messages = reacting to edges

This is the whole reason the state machine emits messages. move_player doesn't run its launch logic every frame — it runs it only on the frame a Dashing transition arrives in its MessageReader. Several systems in dash.rs read that same message independently: one launches, one swaps collision groups on, one resets rotation when the dash ends, one spawns the landing effect. Each stays tiny and single-purpose.

Toggling what can hit what

During a dash the player's normal hitbox turns off (you can't die) while a dash collider turns on (you can kill). Two symmetric systems flip CollisionGroups on the transition in and back out:

fn toggle_player_collider(
    mut q: Query<&mut CollisionGroups, With<PlayerCollider>>,
    mut ev_changed_state: MessageReader<PlayerChangedState>,
) {
    let mut groups = match q.single_mut() { Ok(r) => r, Err(_) => return };
    for ev in ev_changed_state.read() {

🦀 Rust concept · .chain() for deterministic order

The dash plugin registers all ten of its systems with .chain(), forcing them to run in the listed order every frame. That matters here: the launch, the collider toggles, and the effect spawns all read the same transition message and must see a consistent world. When several systems share state and order changes the result, .chain() makes it predictable instead of relying on Bevy's parallel scheduler.

Hook, slide & reflect

The signature move: fling a hook, slide to where it lands, and bat enemy arrows back as kills.

The hook is what makes insta-kill feel like its own game. Press shift and a hook sweeps out in front of you; where it connects becomes a target you slide to at high speed. It's built from the same parts you already know — a state, a transition message, a child collider — plus one new twist: an animation played in reverse.

Enter the Hooking state

fn trigger_hook(player_input: Res<PlayerInput>, mut q_player: Query<&mut Player>) {

fn spawn_hooks(
    mut commands: Commands,
    assets: Res<GameAssets>,
    mouse_coords: Res<MouseWorldCoords>,
    mut q_player: Query<(Entity, &Transform, &mut Sprite), With<Player>>,
    mut ev_changed_state: MessageReader<PlayerChangedState>,
) {
    let (player_entity, transform, mut sprite) = match q_player.single_mut() {
        Ok(r) => r, Err(_) => return,

        if ev.new_state != PlayerState::Hooking { continue; }

        animator.play(assets.player_hook_animations[0].clone());

        let dir = (mouse_coords.0 - transform.translation.truncate()).normalize_or_zero();

        let collider = commands.spawn((
            PlayerHookCollider, Sensor, Collider::ball(15.0),
            CollisionGroups::default(),
            Transform::from_translation(-ROT_OFFSET),

    mut commands: Commands,

// When the throw-out animation finishes, play it backwards (retract)
// and drop the collider so it can't keep grabbing on the way back.
fn reverse_animations(
    mut commands: Commands,
    mut q_hooks: Query<&mut AnimationPlayer2D, With<PlayerHook>>,
    q_hook_colliders: Query<Entity, With<PlayerHookCollider>>,
) {
    for mut animator in &mut q_hooks {
        if animator.is_finished() && !animator.is_reverse() {
            animator.reverse();
            for entity in &q_hook_colliders {
                commands.entity(entity).despawn();
            }
        }
    }
}

// Slide the collider along the hook's local X, forward or backward.
fn move_hook_colliders(
    time: Res<Time>,
    q_hooks: Query<&AnimationPlayer2D>,
    mut q_colliders: Query<(&ChildOf, &mut Transform), With<PlayerHookCollider>>,
) {
    for (parent, mut transform) in &mut q_colliders {
        if let Ok(animator) = q_hooks.get(parent.parent()) {
            let sign = if animator.is_reverse() { -1.0 } else { 1.0 };
            transform.translation += sign * Vec3::X * COLLIDER_SPEED * time.delta_secs();
        }
    }
}

The slide

When the hook connects it sets hook_target_pos, and the player flips to Sliding. Back in movement.rs (from chapter 6) a second movement system takes over, driving the body toward that target until it's close enough:

src/player/movement.rsslide
const SLIDE_SPEED: f32 = 1200.0;
const HOOK_SLIDE_DISTANCE: f32 = 90.0;

fn slide_player(mut q_player: Query<(&Transform, &mut Velocity, &mut Player)>) {
    let (transform, mut velocity, mut player) = match q_player.single_mut() {
        Ok(r) => r, Err(_) => return,
    };
    if player.state != PlayerState::Sliding { return; }

    // Close enough? Stop sliding.
    if (transform.translation + PLAYER_HITBOX_OFFSET).truncate()
        .distance_squared(player.hook_target_pos) <= HOOK_SLIDE_DISTANCE.powi(2) {
        player.state = PlayerState::Idling;
    }

    let dir = (player.hook_target_pos - transform.translation.truncate()).normalize_or_zero();
    velocity.linvel = dir * SLIDE_SPEED;
}
🦀 Rust concept · distance_squared & avoiding sqrt

To ask "am I within 90 units?" the code compares distance_squared(...) against 90.0.powi(2) rather than distance(...) against 90.0. Both answer the same question, but skipping the square root is cheaper — and in a hot per-frame check that habit adds up. You'll see squared-distance comparisons throughout the enemy code too.

Keeping the fight around the player

Enemies spawn just off-screen. But if the player runs away, far enemies would be wasted work — so redeploy_enemies teleports any enemy that drifts too far back to the edge of the action. It's cheaper than despawning and respawning, and keeps pressure constant.

pub const SPAWN_OFFSET: f32 = 900.0;
const MAX_PLAYER_DISTANCE: f32 = 1200.0;

fn redeploy_enemies(
    q_player: Query<&Transform, With<Player>>,
    mut q_enemies: Query<&mut Transform, (With<Enemy>, Without<Player>)>,
) {
    let player_transform = match q_player.single() { Ok(r) => r, Err(_) => return };

    for mut enemy_transform in &mut q_enemies {
        if enemy_transform.translation.truncate()
            .distance_squared(player_transform.translation.truncate())
            >= MAX_PLAYER_DISTANCE.powi(2)
        {
            // Snap it back to the spawn ring, on the side facing the player.
            let dir = (player_transform.translation - enemy_transform.translation)
                .truncate().normalize_or_zero();
            enemy_transform.translation =
                player_transform.translation + dir.extend(0.0) * SPAWN_OFFSET;
        }
    }
}

How weapons actually kill

Now the collision file (src/enemy/collision.rs). Rapier emits a CollisionEvent whenever two colliders touch. Each weapon has a system that reads those events, checks "was one side an enemy and the other side my weapon?", and if so flags the enemy. Here's the strike version — the template for all five:

src/enemy/collision.rsstrike kills
use bevy::prelude::*;
use bevy_rapier2d::prelude::*;
use crate::player::strike::Strike;
use super::Enemy;

fn player_strike_collisions(
    q_strikes: Query<&Strike>,
    mut q_enemies: Query<&mut Enemy>,
    q_colliders: Query<&ChildOf, (With<Collider>, Without<Enemy>, Without<Strike>)>,
    mut ev_collision_events: MessageReader<CollisionEvent>,
) {
    for ev in ev_collision_events.read() {
        // We only care about the moment two colliders start touching.
        let (source, target) = match ev {
            CollisionEvent::Started(a, b, _) => (a, b),
            CollisionEvent::Stopped(..) => continue,
        };

        // Colliders are children; hop to their owning entities.
        let source_parent = match q_colliders.get(*source) { Ok(p) => p.parent(), Err(_) => continue };
        let target_parent = match q_colliders.get(*target) { Ok(p) => p.parent(), Err(_) => continue };

        // Whichever side is the enemy...
        let mut enemy = if let Ok(r) = q_enemies.get_mut(source_parent) { r }
            else if let Ok(r) = q_enemies.get_mut(target_parent) { r }
            else { continue };

        // ...only counts if the OTHER side is a strike.
        if q_strikes.get(source_parent).is_err() && q_strikes.get(target_parent).is_err() {
            continue;
        }
        enemy.disabled = true;   // dead.
    }
}

use rand::{thread_rng, Rng};
use bevy::prelude::*;
use bevy_rapier2d::prelude::*;
use crate::animation::AnimationPlayer2D;
use crate::{enemy::spawn::SPAWN_OFFSET, player::Player, world::camera::YSort, GameAssets, GameState};

fn spawn_enemies(
    mut commands: Commands,
    assets: Res<GameAssets>,
    enemy_spawn_cooldown: Res<EnemySpawnCooldown>,
    q_player: Query<&Transform, With<Player>>,
) {
    let player_transform = match q_player.single() { Ok(r) => r, Err(_) => return };
    if !enemy_spawn_cooldown.timer.just_finished() { return; }

    let mut rng = thread_rng();
    // A random point on the spawn circle: rotate the X axis by a random angle.
    let pos = player_transform.translation
        + Quat::from_rotation_z(rng.gen_range(0.0..TAU)).mul_vec3(Vec3::X) * SPAWN_OFFSET;

    let mut animator = AnimationPlayer2D::default();
    animator.play(assets.enemy_bat_animations[0].clone()).repeat();

    let collider = commands.spawn((
        Collider::ball(8.0),
        ActiveEvents::COLLISION_EVENTS,          // so weapon hits register (ch12)
        CollisionGroups::default(),
        Transform::from_translation(Vec3::new(0.0, -5.0, 0.0)),
    )).id();

    let shadow = commands.spawn((
        YSort(-1.0),
        Sprite::from_image(assets.enemy_bat_shadow.clone()),
        Transform::from_translation(Vec3::new(0.0, -23.0, 0.0)),
    )).id();

    commands.spawn((
        Enemy { score: SCORE, ..default() },
        EnemyBat,
        YSort(0.0),
        animator,
        Sprite { image: assets.enemy_bat.clone(),
                 texture_atlas: Some(TextureAtlas::from(assets.enemy_bat_layout.clone())),
                 ..default() },
        Transform::from_translation(pos).with_scale(Vec3::splat(2.0)),
    )).add_children(&[shadow, collider]);
}

#[derive(Debug, Default, PartialEq, Clone, Copy)]

#[derive(Component)]
pub struct EnemyArcher {

    moving_cooldown: Timer,

impl Default for EnemyArcher {
    fn default() -> Self {
        // Start with the cooldown already elapsed, so it can act immediately.
        let mut moving_cooldown = Timer::from_seconds(MOVING_COOLDOWN, TimerMode::Once);
        moving_cooldown.set_elapsed(Duration::from_secs_f32(MOVING_COOLDOWN));
        Self { state: ArcherState::default(), moving_cooldown }
    }

    for (archer, mut animator) in &mut q {

            ArcherState::Idling   => assets.enemy_archer_animations[0].clone(),
            ArcherState::Moving   => assets.enemy_archer_animations[1].clone(),
            ArcherState::Shooting => assets.enemy_archer_animations[2].clone(),
            ArcherState::Stunned  => assets.enemy_archer_animations[3].clone(),
        };
        animator.play(animation).repeat();
    }
}

fn trigger_shooting(
    q_player: Query<&Transform, With<Player>>,
    mut q_archers: Query<(&Transform, &mut EnemyArcher), Without<Player>>,
) {
    let player_pos = match q_player.single() { Ok(r) => r.translation, Err(_) => return };

    for (archer_transform, mut archer) in &mut q_archers {
        if archer.state == ArcherState::Shooting || archer.state == ArcherState::Stunned { continue; }
        if !archer.moving_cooldown.is_finished() { continue; }

        if archer_transform.translation.distance_squared(player_pos) <= SHOOT_RANGE.powi(2) {
            archer.state = ArcherState::Shooting;
        }
    }
}

CHAPTER 16

Infinite LDtk world

Stream an endless tiled map in chunks around the player, choosing each chunk deterministically from a seed.

So far the player floats over a flat background color. Now we lay down real terrain — but the world is infinite, so we can't spawn it all. Instead the game streams a grid of chunks around the player: spawn the ones coming into view, despawn the ones left behind, and pick each chunk's layout from a seeded random so the same seed always makes the same world. Tiles are authored in LDtk, a free 2D level editor, and loaded with the bevy_ecs_ldtk crate.

🦀 Rust concept · Seeded vs thread-local RNG

// The IIDs of the hand-made LDtk levels we pick between (copy them from level.ldtk).
const IIDS: [&str; 6] = [
    "4561cae1-8990-11ee-bdb7-27b92e7f0bd1",
    "4c5c13d0-8990-11ee-bb97-5335be5f091d",
    // ...four more level iids...
];

#[derive(Component)]
pub struct Chunk { x: i32, y: i32 }

fn map_indices_to_world_coords(x: i32, y: i32) -> Vec3 {
    Vec3::new(x as f32 * CHUNK_SIZE, y as f32 * CHUNK_SIZE, -BACKGROUND_ZINDEX_ABS)
}

fn world_coords_to_map_indices(pos: Vec3) -> (i32, i32) {
    let x = (pos.x / CHUNK_SIZE) as i32 + if pos.x < 0.0 { -1 } else { 0 };
    let y = (pos.y / CHUNK_SIZE) as i32 + if pos.y < 0.0 { -1 } else { 0 };
    (x, y)
}

// Turn (chunk x, chunk y, world seed) into a stable index into our layouts.
fn map_indices_to_index(x_index: i32, y_index: i32, seed: u32) -> usize {
    let seed = seed as u64 + x_index.unsigned_abs() as u64 + y_index.unsigned_abs() as u64;
    let mut rng = GameRng::seed_from_u64(seed);   // GameRng = Xoshiro256PlusPlus
    rng.gen_range(0..IIDS.len())
}

fn level_set_from_map_indices(x: i32, y: i32, seed: u32) -> LevelSet {
    LevelSet::from_iids([IIDS[map_indices_to_index(x, y, seed)]])
}

Streaming chunks around the player

Each frame, compute which chunk indices should be loaded (a rectangle around the player), despawn any loaded chunk outside it, and spawn any missing one inside it:

src/world/map.rsstream
fn adjust_chunks(
    mut commands: Commands,
    assets: Res<GameAssets>,
    seed: Res<Seed>,
    q_player: Query<&Transform, With<Player>>,
    q_chunks: Query<(Entity, &Chunk)>,
) {
    let Ok(player_transform) = q_player.single() else { return; };
    let player_pos = player_transform.translation;

    let max_index = world_coords_to_map_indices(player_pos + Vec3::new( CAMERA_SIZE_X,  CAMERA_SIZE_Y, 0.0));
    let min_index = world_coords_to_map_indices(player_pos + Vec3::new(-CAMERA_SIZE_X, -CAMERA_SIZE_Y, 0.0));

    // Despawn chunks that fell outside the window; remember which remain.
    let mut existing = Vec::new();
    for (entity, chunk) in &q_chunks {
        if chunk.x >= min_index.0 && chunk.x <= max_index.0
            && chunk.y >= min_index.1 && chunk.y <= max_index.1 {
            existing.push((chunk.x, chunk.y));
        } else {
            commands.entity(entity).despawn();
        }
    }

    // Spawn any chunk in range we don't have yet.
    for i in min_index.0..=max_index.0 {
        for j in min_index.1..=max_index.1 {
            if existing.contains(&(i, j)) { continue; }
            commands.spawn((
                Chunk { x: i, y: j },
                LdtkWorldBundle {
                    transform: Transform::from_translation(map_indices_to_world_coords(i, j)),
                    ldtk_handle: LdtkProjectHandle { handle: assets.level.clone() },
                    level_set: LevelSet::from_iids(level_set_from_map_indices(i, j, seed.0)),
                    ..default()
                },
            ));
        }
    }
}

const MAIN_VOLUME_DELTA: f64 = 0.05;

#[derive(Resource)]
pub struct GameAudio { pub main_volume: f64 }

impl Default for GameAudio {
    fn default() -> Self { Self { main_volume: 0.5 } }
}

impl GameAudio {
    pub fn update(&mut self, x: f64) {
        self.main_volume = (self.main_volume + x).clamp(0.0, 1.0);
    }
}

// Human volume is linear (0..1); audio hardware wants decibels (logarithmic).
pub fn amplitude_to_decibels(amplitude: f64) -> Decibels {
    if amplitude <= 0.0 { return Decibels::SILENCE; }
    Decibels(20.0 * (amplitude as f32).log10())
}

// Mouse wheel nudges the master volume (input from ch5).
fn update_main_volume(player_input: Res<PlayerInput>, mut game_audio: ResMut<GameAudio>) {
    if player_input.scroll == 0.0 { return; }
    game_audio.update(-player_input.scroll as f64 * MAIN_VOLUME_DELTA);
}

The reusable play-sound message

Rather than each feature calling the audio engine directly, they describe a sound and send it. The listener handles volume, looping, reverse, and — the nice touch — a random pitch offset so the same clip fired rapidly (like the strike chain) stays lively:

src/audio/sound.rsPlaySound
use rand::{thread_rng, Rng};
use bevy::prelude::*;
use bevy_kira_audio::prelude::{AudioSource, *};
use super::{amplitude_to_decibels, GameAudio};

#[derive(Message)]
pub struct PlaySound {
    pub clip: Handle<AudioSource>,
    pub volume: f64,
    pub playback_rate: f64,
    pub rand_speed_intensity: f64,   // 0.0 = exact pitch; >0 = jitter it
    pub repeat: bool,
    pub parent: Option<Entity>,      // Some(e) => spatial sound at that entity
}

// Default lets callers set only the fields they care about (ch1's ..default()).
impl Default for PlaySound {
    fn default() -> Self {
        Self {
            clip: Handle::default(), volume: 1.0, playback_rate: 1.0,
            rand_speed_intensity: 0.0, repeat: false, parent: None,
        }
    }
}

fn play_sounds(
    audio: Res<Audio>,
    game_audio: Res<GameAudio>,
    mut ev_play_sound: MessageReader<PlaySound>,
) {
    let mut rng = thread_rng();
    for ev in ev_play_sound.read() {
        let speed_offset = if ev.rand_speed_intensity == 0.0 { 0.0 }
            else { rng.gen_range(-1.0..1.0) * ev.rand_speed_intensity };

        audio.play(ev.clip.clone())
            .with_volume(amplitude_to_decibels(ev.volume * game_audio.main_volume))
            .with_playback_rate(ev.playback_rate + speed_offset);   // the pitch wobble
    }
}

pub struct GameSoundPlugin;

impl Plugin for GameSoundPlugin {
    fn build(&self, app: &mut App) {
        app.add_message::<PlaySound>()
            .init_resource::<GameAudio>()
            .add_systems(Update, play_sounds);
    }
}

fn animate_world_texts(
    time: Res<Time>,
    mut q: Query<(&mut Transform, &mut TextColor, &mut WorldText)>,
) {
    for (mut transform, mut text_color, mut world_text) in &mut q {
        world_text.timer.tick(time.delta());
        let t = world_text.timer.elapsed_secs() / world_text.timer.duration().as_secs_f32();

        // Sample the curves at normalized time t (0..1).
        transform.scale = world_text.scale_curve.position(t).y
            * Vec3::splat(world_text.font_scale / FONT_SCALE_RATIO);
        let c = world_text.font_color.to_srgba();
        text_color.0 = Color::srgba(c.red, c.green, c.blue, world_text.alpha_curve.position(t).y);
    }
}

CHAPTER 19

Game over & restart

Close the loop with the full state machine: die, show the score, reset, and drop straight back into a fresh run.

Time to make death mean something and give the player a way back. This is where the GameState enum from chapter 3 finally uses all four variants: Gaming → GameOver → Restart → Gaming. Each transition just runs OnEnter/OnExit systems — no tangled teardown code, because every feature already cleans up its own entities on the right state edge.

AssetLoading → Gaming → (you die) GameOver
GameOver → (press R/Esc) Restart → (reset + 0.2s) Gaming

The whole game loop is four states and the edges between them.

Dying triggers GameOver

Recall from chapter 6's spawn file: when the player's disabled flag is set (a bat touched the hitbox), a tiny system flips the state. That's all it takes:

🕹 Bevy concept · NextState & state-scoped cleanup

You never change state directly; you queue it with NextState<GameState>, and Bevy applies it, firing every OnEnter/OnExit system for the transition. Because the HUD despawns OnExit(Gaming), the player despawns OnEnter(GameOver), and enemies despawn OnEnter(Restart), teardown is distributed — each feature owns its own lifecycle. No central "reset everything" function to forget to update when you add a feature. This is the quiet superpower of building on states from day one.

fn restart(time: Res<Time>, mut restart_timer: ResMut<RestartTimer>,
           mut next_state: ResMut<NextState<GameState>>) {
    restart_timer.tick(time.delta());
    if restart_timer.just_finished() {
        next_state.set(GameState::Gaming);       // back into the fight
    }
}

pub struct RestartPlugin;

impl Plugin for RestartPlugin {
    fn build(&self, app: &mut App) {
        app.init_resource::<RestartTimer>()
            .add_systems(OnEnter(GameState::Restart), reset_resources)
            .add_systems(Update, restart.run_if(in_state(GameState::Restart)));
    }
}

🦀 Rust concept · cfg — platform-specific code

Some things only exist off the web. The real project gates desktop-only systems (fullscreen toggle, screenshots) with #[cfg(not(target_arch = "wasm32"))], and adds a WASM-only dependency in Cargo.toml under [target.'cfg(target_arch = "wasm32")'.dependencies] (the getrandom crate with its "js" feature, so rand works in a browser). cfg is Rust's compile-time "if this platform" switch — code behind a false cfg isn't compiled at all.

A size-tuned release profile

WASM downloads over the network, so binary size matters more than on desktop. The project adds a dedicated profile that optimizes hard for size:

Cargo.tomlwasm profile
# https://github.com/rust-lang/cargo/issues/4897
[profile.wasm]
inherits = "release"
panic = "abort"      # no unwinding machinery in the binary
opt-level = 'z'      # optimize for size, not speed
lto = true           # link-time optimization: strip cross-crate dead code
codegen-units = 1    # slower compile, smaller/faster output
